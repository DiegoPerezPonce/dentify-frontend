- Route Guards by Roles (RBAC) #4
- Description: Exactly two security roles: ROLE_ADMIN and ROLE_USER (Symfony-style). Product mapping: ROLE_USER = alumno (estudiante / personal clínico en la simulación); ROLE_ADMIN = supervisor-administrador (misma persona en JWT, etiqueta UX “admin” o “supervisor”). Use case rule: ROLE_ADMIN inherits all ROLE_USER clinical capabilities—same routes and screens for pacientes, agenda, radiografías, etc., plus exclusive administration (user CRUD #15, role assignment ADMIN vs USER only, dentists #16, boxes #17, protocols #18, stock configuration #14 + material catalog #26, …). There is NO “switch to student mode” or second app entry: a single admin shell with sections General (clinical, shared with students) + Administración (admin-only); optionally collapse the Administración block in the UI for focus, but that is cosmetic only, not a security boundary. Route policy: clinical/patient-care routes allow ROLE_USER OR ROLE_ADMIN; administration-only routes require ROLE_ADMIN. Only /login is public (no JWT). No public signup (#15). Implement authGuard + roleGuard and shell menu from JWT claims. Include placement for avisos / notas pedagógicas published by ROLE_ADMIN and shown to ROLE_USER (students must see them), e.g. banners on dashboard or patient flows—content may be static at first or backed by API (see backend notices issue).
- Branch: feature/4-route-guards-by-roles-rbac
